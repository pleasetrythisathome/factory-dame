# Feedback: Skip ticketing / process overhead on this project

## The preference

Scarlet explicitly pushed back on formal ticketing for factory-dame / the neurodynamics engine. She wants to keep this project fun and action-oriented, not task-managed.

**Her words (2026-04-16)**:
> "meh maybe we should skip backlog - this project is supposed to be a fun side project and task management feels like work. just keep moving."

Context: I had offered to create ~18 tickets in Backlog.md covering viz polish, Tier 2 (two-layer pulse network), Tier 3 (perceptual features), Tier 1 (live audio), and Tier 4 (consumer integrations). She declined and asked to just proceed.

## Why this fits the bigger picture

This project is explicitly positioned as a **creative counterweight to aswritten** (`Narrative_CreativeCounterweightToAswritten`): breathing room from the "dumber and faster" pragmatism that running a business demands. Aswritten needs process and velocity tracking; factory-dame needs the opposite — freedom to follow curiosity without the overhead that signals "work."

The `backlog/` directory exists (scaffolded via Backlog.md with config.yml and dir structure), but she prefers keeping it unused here. In-session task lists during active building are fine (we used TaskCreate throughout this session to track features in-flight) — the objection is specifically to creating persistent tickets for future work.

## How to apply this going forward

- **Do NOT propose** creating Backlog.md tickets, GitHub issues, or other persistent task-management artifacts for this project unless Scarlet asks for one explicitly.
- **In-session TaskCreate is still welcome** when actively juggling multiple features during a build — it's a working-memory tool, not a planning artifact.
- **At session boundaries**, summarize in conversation ("what's done / what's next") rather than creating tickets. If the thread through future sessions needs to survive, put it in an aswritten memory, not a backlog.
- **This does NOT apply to aswritten or business projects** — for those, structured task management is still appropriate. This is specifically about the creative side project.

- Extends `Narrative_CreativeCounterweightToAswritten` (the strategy behind this project)
- Consistent with `Narrative_AIWorkflowSandbox` — the project itself is a space for exploring how to collaborate creatively with AI tools, and that includes discovering what rituals are welcome vs. unwelcome
